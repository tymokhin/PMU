# NOTES — провали (gaps) Tracealyzer-стріму при heap CycloneTCP у SDRAM

Дата: 2026-06-26
Статус: **ВИРІШЕНО**

---

## Симптом

При `#define NET_MEM_POOL_SUPPORT DISABLED` (CycloneTCP бере NetBuffer'и з кучі
heap_5 у зовнішній SDRAM @ `0xC0000000`) у потоці трасування Tracealyzer на десктоп
по TCP/IP (порт 12000) зʼявляються **провали — gaps**: десктоп періодично втрачає
блоки подій, на таймлайні видно «дірки».

З `NET_MEM_POOL_SUPPORT ENABLED` (статичний пул `memPool[64][1536]` у внутрішньому
AXI SRAM) проблеми немає. Сама SDRAM протестована й справна.

---

## Root cause

**Першопричина — крихкий конфіг TCP-стрімінгу, а НЕ пам'ять.**

Початкова гіпотеза «повільна 16-бітна SDRAM не тягне смугу» виявилась **хибною**.
Реальний потік трас ≈ **74 КБ/с** (57 КБ за 2.67 с прогону) — мізер для 100 Мбіт лінка
й для SDRAM. Вирішальний контр-доказ: при збільшенні TX-буфера стало **гірше**, а не
краще — пряма ознака, що вузьке місце не в пропускній здатності.

Аналіз `.psf`-логу (втрати рахуються за розривами наскрізного поля `EventCount` у
`BaseEvent`) виявив **два класи провалів**:

1. **~40–46 мс паузи (≈60 % втрат)** — класичний **Nagle (на пристрої) × delayed-ACK
   (на десктопі)** deadlock на крихітному 576-Б TX-буфері: відправник тримає малий
   сегмент до приходу ACK, приймач відкладає ACK ~40 мс → потік завмирає пачками по 40 мс.
2. **~0.5–1 с паузи (≈40 % втрат)** — **TCP RTO**: при поодинокій реальній втраті пакета
   й дефолтному CycloneTCP `TCP_MIN_RTO = TCP_INITIAL_RTO = 1000 мс` відправник чекає
   цілу секунду до ретрансмісії. Крихітний 5-КБ буфер рекордера (10 сторінок × 500 Б)
   за цей час переповнюється → події відкидаються (`DroppedEventCounter`).

### Чому проявилось лише з SDRAM, а з пулом у SRAM — ні

Конфіг TCP-стрімінгу був **на межі стабільності**: Nagle увімкнено, SACK вимкнено,
TX-буфер 576 Б, RTO 1 с. Швидкий пул у внутрішньому SRAM **маскував** цю крихкість —
стек встигав зливати сторінки вчасно, межа переповнення не досягалась.

SDRAM-куча додає рівно стільки латентності й джитеру, щоб **виштовхнути крихкий конфіг
за межу**:
- доступ до 16-бітної SDRAM повільніший за AXI SRAM;
- `pvPortMalloc`/`vPortFree` (heap_5) працюють під `vTaskSuspendAll` із best-fit
  пошуком по зв'язному списку, що фізично лежить у SDRAM → нерівномірні призупинення
  планувальника;
- зростає ймовірність поодиноких реальних втрат пакета.

Цього вистачає, щоб з'явились delayed-ACK стопи й поодинокі RTO, які лавиноподібно
валять крихітний трас-буфер. Тобто SDRAM — **тригер**, а не причина; причина — крихкий
транспорт.

### Чому більший TX-буфер робив гірше

`TCP_SACK_SUPPORT` був вимкнений (0). Без SACK будь-яка втрата → **Go-Back-N**:
ретрансмісія всього вікна після RTO. Більше вікно = більше даних «в польоті» = більше
ретрансмісій + head-of-line blocking + RTO backoff (1→2→4 с) → throughput колапсує до нуля.
Збільшення буфера без SACK — рух у неправильний бік.

## Рішення (зміни A–D)

Лікували **транспортний рівень**, а не пам'ять — бо смуга/швидкість SDRAM ніколи не були
вузьким місцем.

| | Зміна | Файл | Чому |
|---|---|---|---|
| A | `setsockopt(IPPROTO_TCP, TCP_NODELAY)` на трас-сокеті | `Tracealyzer4/streamports/trcStreamingPort.c` (`trcSocketAccept`) | вимикає Nagle → прибирає ~40 мс delayed-ACK стопи (≈60 % втрат). Найдешевше, найбільший виграш |
| B | `TCP_SACK_SUPPORT 1` | `Conf/CycloneTCP/net_config.h` | дешеве відновлення після поодинокої втрати; усуває ефект «більший буфер = гірше» |
| C | `TCP_INITIAL_RTO 500`, `TCP_MIN_RTO 200` | `Conf/CycloneTCP/net_config.h` | на LAN RTT субмілісекундний → секундна RTO-пауза скорочується до ~0.2 с |
| D | `TCP_DEFAULT_TX_BUFFER_SIZE` 576 → 4096 | `Conf/CycloneTCP/net_config.h` | нормальне пайплайнування (безпечно лише разом із SACK) |

## Результат (той самий парсер `.psf`)

| Метрика | ДО (A–D) | ПІСЛЯ (A–D) |
|---|---|---|
| Втрачено подій | **71.2 %** (10813/15189) | **0.9 %** (290/32479) |
| Сплесків провалів | 96 | 3 |
| Великі RTO-паузи (≥300 подій) | 7 (до 591 мс) | **0** |
| Найбільший провал | 727 подій / 591 мс | 216 подій / 182 мс |
| Розмір трейсу за прогін | 57 КБ | 419 КБ |

Великі секундні RTO-стопи **зникли повністю**. Втрати подій впали з **71.2 % → 0.9 %**
**без перенесення мережевих буферів у внутрішнє SRAM** — логер успішно працює з кучею
CycloneTCP у SDRAM. Лишилось 3 дрібні інциденти (~0.9 %), найбільший ~182 мс — поодинока
реальна втрата пакета, тепер швидко відновлена коротшим RTO + SACK.

## Якщо потрібен «майже нуль» (опційно, наразі не потрібно)

Залишкові ~0.9 % — це поодинокі **реальні** втрати пакетів, імовірно пов'язані з SDRAM
(латентність/джитер або взаємодія D-cache write-back ↔ SDRAM). Для подальшого зменшення:

- Перевірити `trcStreamDiagnostics.retransmitCount`/`rto` у дебагері + Wireshark
  («checksum incorrect» на сегментах від пристрою → корупція даних у SDRAM).
- A/B: зробити MPU-регіон SDRAM **non-cacheable** і порівняти (проста memtest такого не ловить).
- Або винести **лише мережевий heap у внутрішнє SRAM** — heap_5 підтримує кілька регіонів,
  app-дані лишаються в SDRAM. Для поточної мети (логер працює з SDRAM) це вже не потрібно.

## Опорні факти про конфігурацію (для довідки)

- MCU: **STM32H725AGIx** (Cortex-M7), STM32CubeIDE/GCC.
- SDRAM **MT48LC32M16** на FMC @ `0xC0000000`, шина **16-біт**, CAS=3.
- MPU-регіон SDRAM: **cacheable + bufferable, write-back, non-shareable**.
- FreeRTOS **heap_5**, єдиний регіон кучі повністю в SDRAM @ `0xC0000000`;
  `osAllocMem()` → `pvPortMalloc()` → heap_5 → SDRAM.
- Перемикач алокатора NetBuffer: `Conf/CycloneTCP/net_config.h`
  (`ENABLED` → пул `memPool[64][1536]` у `.bss`/AXI SRAM; `DISABLED` → SDRAM heap).
- ETH DMA-буфери (дескриптори + `txBuffer`/`rxBuffer` драйвера) — у **non-cacheable
  RAM_D2 @ 0x30000000**; драйвер memcpy'ить між NetBuffer і DMA-буферами, тож
  DMA-когерентність забезпечена розміщенням, а **не** ручним обслуговуванням кешу.
  Це підтверджує, що провали — **не** проблема когерентності/корупції, а транспорт.
- Трас-буфер рекордера: 10 сторінок × 500 Б = **5 КБ**.
- Tracealyzer стрім: TCP порт **12000**.

Парсер логу: `/tmp/parse_psf.py` + `/tmp/analyze*.py` (тимчасові, на машині розробника).
